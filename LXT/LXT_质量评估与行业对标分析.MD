# LXT 质量评估与行业对标分析

> 更新时间：2026-04-28  
> 基于 2026 年 AI 短剧行业现状，对 LXT 模式资产库、脚本生成、成片三阶段进行质量对标评估。
> **实测验证**：2026-04-28 使用三只小猪故事对"测试LXT"项目进行了完整流水线测试（剧本→分镜→资产库→图片生成），下文包含实测数据对比。

---

## 一、评估范围

| 阶段 | 评估内容 | 对标行业基准 |
|------|---------|------------|
| 资产库（lxt-assets） | 角色/场景/道具提取质量、档案完整度 | OpenCharacter、Seedance 2.0 角色管理、纳米漫剧流水线 |
| 脚本生成（lxt-script/storyboard/final-script） | 剧本改编、分镜拆解、制作脚本质量 | DramaBench、DSR 双阶段精炼、Camera Artist、Dramaturge |
| 成片（lxt-final-film） | 图片生成、视频生成、镜头连续性 | Seedance 2.0 工作流、FilmWeaver、行业首尾帧实践 |

## 二、资产库阶段（lxt-assets）

### 2.1 当前架构

- **三路并行 LLM 分析**：`NP_AGENT_CHARACTER_PROFILE` + `NP_SELECT_LOCATION` + `NP_SELECT_PROP`
- **双源输入**：`novelText` + `shotListContent`（上限 30,000 字）
- **去重机制**：已提取资产名单注入 Prompt，复合唯一键 `(lxtProjectId, kind, name)`
- **用户交互**：LLM 分析 → 人工确认档案 → 一键批量确认

### 2.2 达标项（符合/超越 2026 行业标准）

| 维度 | LXT 现状 | 评价 |
|------|---------|------|
| 角色字段丰富度 | 15+ 结构化字段：`role_level`(S/A/B/C/D)、`archetype`(20+ 原型)、`personality_tags`(10+ 标签)、`era_period`、`social_class`、`occupation`、`costume_tier`(1-5)、`suggested_colors[]`、`primary_identifier`、`visual_keywords[]`、`gender`、`age_range`、`body_proportion`、`species_traits[]` | **接近** OpenCharacter 标准（20+ 字段） |
| 关系提取 | 9 种关系类型（superior/friendly/hostile/romantic/family/ally/rival/competing/other）+ 方向性（单向/双向）+ 描述 | **达标** |
| 道具提取保守度 | 9 条严格排除规则 + 可携带性测试 + 不可替代性测试 + 跨场景复发证据；"宁可漏掉"原则 | **达标** |
| 场景空间锚定 | 3 个宽镜头描述（前景/中景/背景）+ 2-6 个可放置角色插槽 + 至少 3 个稳定锚点对象 | **达标** |
| 非人类角色 | 专属 `body_proportion` + `species_traits`（3-8 关键词） | **超越** 行业通用标准 |
| 人称视角映射 | `introduction` 强制处理"我→主角"等叙事视角转换 + 关系图谱 + 称呼别名 | **超越** |

### 2.3 差距项

| 差距 | 严重度 | 描述 |
|------|--------|------|
| **关系数据未利用** | 高 | `agent_character_profile` 已提取 `relationships[]`，但下游脚本/成片阶段完全不消费。行业做法（StoryWeaver、Camera Artist）将关系图作为跨镜空间定位和互动生成的核心约束 |
| **缺少大五人格** | 中 | `personality_tags` 仅覆盖中文网文语境标签（高冷、温柔、腹黑），缺少 OCEAN 心理学维度。DuoDrama、OpenCharacter 采用 Big Five 实现跨模型人格迁移 |
| **缺少角色状态追踪** | 中 | 无分镜/分集的角色外观变化管理。Seedance 2.0 角色管理系统支持叙事时间线内追踪年龄、服装、面部特征变化 |
| **多集分析未实现** | 中 | `lxt-analyze-assets` 只取第一集（`findFirst`），多集项目信息不完整。小云雀扫描全部剧本分析角色全生命周期 |
| **无自动质量评分** | 中 | 完全依赖用户人工"确认档案"。GIBO WATCH、纳米漫剧流水线有自动扫描视觉不一致性的质量检查 |

### 2.4 资产图片提示词质量分析

#### 2.4.1 当前架构

资产图片生成是通过 LLM 先推理出视觉描述（`description` 字段），再拼接 Prompt Suffix 和艺术风格后，提交给图像模型生成。

**Prompt 组装流程**（`lxt-asset-image-task-handler.ts`）：
```
[bodyProportion?] + [description] + [artStyle] + [PromptSuffix] → 图像模型
```

**最终 Prompt 结构**：

| 资产类型 | Prompt 结构 | 宽高比 | 使用模型 |
|---------|------------|--------|---------|
| 角色 | `{bodyProportion}，{description}，{artStyle}，角色设定图，画面分为左右两个区域：【左侧区域】占约1/3宽度...【右侧区域】占约2/3宽度，角色三视图...` | 3:2 | `userModels.characterModel` |
| 场景 | `{description}，{artStyle}` | 1:1 | `userModels.locationModel` |
| 道具 | `{description}，{artStyle}，道具设定图，画面分为左右两个区域：【左侧区域】占约1/3宽度...【右侧区域】占约2/3宽度，三视图...` | 3:2 | `userModels.locationModel` |

**description 生成方式**（`confirm-profile/route.ts`）：
- LLM 按严格格式生成 tag 式视觉描述：`物种+体型, , 服装, , 体格, , 脸型, , 眉眼鼻唇, , 发型发色, , 配饰`
- 每个属性必须包含：颜色词 + 材质词 + 形态词
- 使用英文逗号分隔，`", , "`（双逗号）分隔身体区域

#### 2.4.2 达标项

| 维度 | 评价 |
|------|------|
| 角色设计图结构 | **达标**：左侧 1/3 面部特写 + 右侧 2/3 三视图（正面/侧面/背面），纯白背景——此格式与 Seedream 最佳实践一致 |
| 道具三视图格式 | **达标**：左侧主视图 + 右侧三视图，纯白背景，明确禁止手部/桌面/环境 |
| 描述格式 | **达标**：tag 式逗号分隔格式专为扩散模型设计，语义密度高，无冗余修饰 |
| 宽高比映射 | **达标**：角色 3:2、场景 1:1、道具 3:2，通过 `SIZE_MAP_4K` 正确映射到 Seedream 像素尺寸 |

#### 2.4.3 差距项

| 差距 | 严重度 | 描述 |
|------|--------|------|
| ~~**缺少 Negative Prompt**~~ | ✅ 已修复 (P0-6) | 2026-04-28 修复：传入画风特定 negative prompt，Ark/Gemini 模型使用 `convertNegativeToPositivePrompt` 退化路径 |
| ~~**LXT 资产无参考图注入**~~ | ✅ 已修复 (P0-6) | 2026-04-28 修复：角色类型查询同名已有资产的 imageUrl 作为参考图；实测首次生成时 referenceImageCount=0（预期行为） |
| ~~**道具使用场景模型**~~ | ✅ 已修复 (P0-7) | 2026-04-28 修复：道具优先使用 `characterModel`，fallback 到 `locationModel` |
| **描述回退链有风险** | **中** | 回退链 `description → summary → name` 意味着如果 description 为空，会使用 `summary`（叙事性文本，非视觉描述）作为图片 prompt，必然产出低质图像 |
| **中英文混合** | **中** | Prompt Suffix 是中文长文本，artStyle 是英文。例如：`中文视觉描述, 英文风格描述，角色设定图，中文布局指令`。对 Seedream 影响较小（中英双语能力），但对 Imagen/OpenAI 模型可能降低质量 |
| **场景无布局指导** | **中** | `LOCATION_PROMPT_SUFFIX = ''`，场景图无结构性 layout 指导。行业最佳实践建议至少给锚点对象/光线方向/构图提示 |
| **无质量后验** | **中** | 生成后无任何质量验证：不检查是否成功生成三视图布局、不检查面部是否可见、不检查纯白背景 |

## 三、脚本生成阶段（lxt-script → lxt-storyboard → lxt-final-script）

### 3.1 当前架构

- **Stage 1（小说→剧本）**：单次 LLM 调用，格式转换（旁白/对白拆分）
- **Stage 2（剧本→分镜）**：可选两阶段（script_analysis → script_to_storyboard），10 条约束规则
- **Stage 3（分镜→制作脚本）**：BullMQ 异步 + 单镜四阶段并行流水线（Phase1 基础分析 → Phase2A 图像 + Phase2B 表演并行 → Phase3 视频合成），并发度 5

### 3.2 达标项（符合/超越 2026 行业标准）

| 维度 | LXT 现状 | 评价 |
|------|---------|------|
| **摄影机词汇量** | 16 种景别（ELS→ECU 全系列）+ 16 种视角（平视→荷兰角）+ 20+ 种运动（4 大类）+ 微移模板库 | **超越** Camera Artist 框架标准 |
| **表演指导词汇** | 58 个术语（7 大类：头颈/面部/手臂/躯干/步态/爆发/微表演）；4 帧连续动势衔接规则 | **超越** DuoDrama 标准 |
| **4 帧关键帧架构** | 帧间动势衔接（帧 N 结束 → 帧 N+1 开始）+ 每帧 7 必备要素 + 1 个具体递进变化 | **匹配** Seedance 2.0 多参考能力 |
| **场景类型系统** | 5 种（daily/emotion/action/epic/suspense），每种独立摄影+表演+视频策略 | **达标** |
| **空间上下文** | 每镜追踪 scene/time/lighting/camera_direction/characters 位置/framing/scene_type | **接近** Script-a-Video Stream Factorization |
| **中文 Prompt** | Phase2A 169 行、Phase3 168 行，包含微移模板库、光线克制规则、全局空间约束 | **优势** |
| **阶段并行** | Phase2A+2B 并行，5 并发 | **达标** |

### 3.3 差距项

| 差距 | 严重度 | 描述 |
|------|--------|------|
| **无多智能体协作** | 高 | 每 Phase 仅一次 LLM 调用，生成即结束。Dramaturge 用 Global→Scene→Revision 三级评审提升 53.2%；Postwriter 用 Plan→Draft(3-5)→Validate(5+10)→Repair(3轮)→Revise |
| **无质量验证闭环** | 高 | 无生成后质量自评。DramaBench 六维评估（格式规范、叙事效率、角色一致性、情感深度、逻辑一致性、冲突处理）可实现自动化 |
| **跨镜上下文缺失** | 中 | Phase 1/2A/2B 每镜分析完全独立，无前后镜信息。Camera Artist RSG 每镜显式推理前一镜状态 |
| **英文 Prompt 不对标** | 中 | 中文版系统性地更详细（Phase2A 169 行 vs 125 行），英文版缺 7 要素模板、微移模板库、空间约束等关键规则。Seedance 2.0 需英文摄影机提示词 |
| **小说→剧本过于保守** | 中 | "绝对禁止添加对话、动作或描述"，仅格式转换。DSR 双阶段精炼先丰富叙事再转换，达人类水平 82.7% |
| **角色关系未注入** | 中 | `buildLxtAssetPromptContext()` 仅注入名称+摘要，未注入关系图谱。导致对话语气、空间定位缺乏关系约束 |

## 四、成片阶段（lxt-final-film）

### 4.1 当前架构

- **图片生成**：4 宫格图像 → sharp 切割 4 张 → 末张设为尾帧
- **视频生成**：首尾帧模式 + 多分镜上下文（±1 镜）+ 资产参考图注入
- **硬编码参数**：8 秒、720p

### 4.2 达标项

| 维度 | LXT 现状 | 评价 |
|------|---------|------|
| 多分镜上下文 | `buildMultiShotVideoPrompt` 前后各 1 镜 storyboard 格式 | **达标** |
| 首尾帧模式 | 自动检测 `videoEndFrameUrl` 切换 `firstlastframe`/`normal` | **达标** |
| 资产参考注入 | 角色/场景/道具图 Base64 归一化注入视频模型 | **达标** |
| 模型适配层 | Strategy 模式：Seedance 2.0/1.5/1.0 + Wan 2.6 + Veo 3.1 + Sora 2 + Kling 3 | **达标** |

### 4.3 差距项

| 差距 | 严重度 | 描述 |
|------|--------|------|
| ~~**4 帧未充分利用**~~ | ✅ 已修复 (P0-1) | 2026-04-28 修复：4 帧全部作为参考图注入 Seedance 2.0，利用时序关系提升运动连贯性 |
| **无帧插值** | 高 | 完全无 VFI。ArbInterp（ICLR 2026）支持 2x-32x 任意倍率插值；EDEN（CVPR 2025）LPIPS 降低 10% |
| ~~**无种子锁定**~~ | ✅ 已修复 (P0-2) | 2026-04-28 修复：同一集共用 videoSeed，保持噪声模式一致 → 角色面部几何跨镜稳定 |
| ~~**无前一镜尾帧注入**~~ | ✅ 已修复 (P0-3) | 2026-04-28 修复：前一镜 videoEndFrameUrl 作为额外参考图传入当前镜视频生成 |
| ~~**参数硬编码**~~ | ✅ 已修复 (P0-4) | 2026-04-28 修复：videoDuration（默认 8s）、videoResolution（默认 720p）从 content 读取，UI 可选 |
| **无音频** | 中 | 完全没有 TTS/音效/BGM 生成。Seedance 2.0 原生支持 8+ 语言音频视频同步 |
| **无变体筛选** | 中 | 每镜仅生成 1 个视频。纳米漫剧流水线一次多选，成功率 90%+ |
| **无转场处理** | 低 | 各镜独立生成，无转场。StoryGen-Atelier "插值链"算法处理过渡 |

### 4.4 四宫格图片提示词质量分析

#### 4.4.1 当前架构

四宫格图片的 prompt 分为两个阶段生成：

**阶段 1 — LLM 推理（Phase 2A）**：
- Prompt 文件：`shot_phase2a_image.zh.txt`（169 行，中文）/ `shot_phase2a_image.en.txt`（125 行，英文）
- 输出格式：`全局物理空间位置:`（≥80 字）+ 4 个 `分镜图片布局:` 子块（每块 7 要素：景别/机位/光线/空间轴/角色姿态/递进变化/镜头高度）
- 总长度约束：500-700 字，每帧 80-120 字

**阶段 2 — 最终 Prompt 组装**（`handleLxtFinalFilmImageTask`）：
```
fullPrompt = gridPrefix + imagePrompt + ", " + artStyleText
```
示例（写实风格）：
```
2x2 grid layout, 4 sequential keyframes, 全局物理空间位置: ...；分镜图片布局：左上图片:...右上图片:...左下图片:...右下图片:..., Realistic cinematic look, real-world scene fidelity, rich transparent colors, clean and refined image quality
```

**阶段 3 — 后处理**：AI 生成宫格图 → `sharp` 等分切割为 4 张 → 上传 COS

#### 4.4.2 达标项

| 维度 | 评价 |
|------|------|
| Phase 2A 中文 Prompt 质量 | **优势**：169 行，23 条约束规则，7 要素模板，光线克制规则，全局空间 ≥80 字，每帧强制 1 个具体递进变化 |
| 网格前缀智能选择 | **达标**：`resolveGridPromptPrefix` 按画风分类（anime → 中文"四格漫画"，其他 → 英文"2x2 grid layout"），避免"漫画"与写实风格的语义冲突 |
| 资产参考图注入 | **达标**：`collectLxtReferenceImages` 按角色→场景→道具优先级排序，按模型能力上限截断，Base64 归一化 |
| 宽高比映射 | **达标**：`videoRatio`（默认 9:16）→ `SIZE_MAP_4K` → Seedream 像素尺寸（3072x5456） |
| 历史保留 | **达标**：`imageSets[]` 保留最近 2 套，支持回退对比 |

#### 4.4.3 差距项

| 差距 | 严重度 | 描述 |
|------|--------|------|
| ~~**缺少 Negative Prompt**~~ | ✅ 已修复 (P0-5) | 2026-04-28 修复：`handleLxtFinalFilmImageTask` 传入画风特定 negative prompt；Ark/Gemini 模型使用 `convertNegativeToPositivePrompt` 退化路径 |
| **中英文混合 Prompt** | **高** | 最终 Prompt 结构为 `英文gridPrefix + 中文Phase2A内容 + 英文artStyle`。Phase 2A 始终输出中文（无论用户语言设置），但 gridPrefix（非 anime 风格时）和 artStyle 是英文。对 Seedream（中英双语）影响较小，对 Imagen/OpenAI（英文优先）模型显著降低质量 |
| **英文 Phase 2A Prompt 不完整** | **中** | 英文版仅 125 行（vs 中文 169 行），缺少 7 要素模板、机位微移模板库、光线克制规则、全局空间约束。当用户使用英文界面时，生成的 imagePrompt 质量系统性低于中文 |
| **4 帧仅用 2 帧** | **中** | 切割后仅 index 0（首帧）和 index 3（尾帧）用于视频生成，index 1、2 完全丢弃。损失的中间帧包含关键的动势递进信息 |
| **切割假设脆弱** | **中** | `splitGridImage` 假设 AI 严格输出 2x2 等分布局，无边框、无间距。若模型生成不规则布局（边框、重叠、非均匀间距），切割会产生错位帧 |
| **无图像质量后验** | **低** | 生成后不验证宫格布局是否成功、4 帧是否完整、分辨率是否符合预期 |

| **Seedream 5.0** | 良好 | 良好 | 静默丢弃 | 低 |
| **Imagen (Google)** | 差 | 良好 | 支持 | 高（英文优先模型） |
| **OpenAI Compatible** | 差 | 良好 | 支持 | 高 |

## 六、关键结论

### 6.1 核心优势

1. **Prompt 工程已达行业一流**：摄影机词汇、表演指导词汇、4 帧架构、中文 Prompt 细节度均超越行业平均水平
2. **4 阶段并行流水线架构合理**：单镜拆分 + 并发处理 + 进度回传，可扩展性强
3. **数据模型设计前瞻**：行级 patch 合并、版本化 JSON、资产唯一键、历史保留均考虑周全
4. **模型适配层成熟**：Strategy 模式统一多供应商，支持 Seedance/Wan/Veo/Sora/Kling

### 6.2 核心短板

1. **缺少质量反馈闭环**：所有阶段均"生成即结束"，没有 Review/Critic/Validate 机制。这是 2026 年行业最显著的代差
2. **连续性约束不足**：跨镜上下文（Phase 1/2A/2B）+ 种子锁定 + 尾帧注入 + 帧插值，四个环节全缺失
3. **数据孤岛**：资产库已提取的关系数据、角色档案等未充分流转到脚本和成片阶段
4. **英文 Prompt 不对标**：中文优势无法传递到需要英文输入的 Seedance 2.0
5. **图片提示词缺少 Negative Prompt**：资产库和四宫格图片生成均不传递 negative prompt，也不使用 Ark 的 positive 退化路径，导致卡通/模糊/低质等常见问题无防护
6. **中英文混合 Prompt 降低非 Seedream 模型质量**：英文 gridPrefix + 中文 Phase2A 内容 + 英文 artStyle 的混合结构对 Imagen/OpenAI 等英文优先模型不友好
7. **LXT 资产缺少参考图一致性保障**：novel-promotion 模式的角色处理程序注入参考图，但 LXT 模式完全不注入

### 6.3 改进优先级矩阵

```
                    低投入          中投入          高投入

高收益    ┌─────────────────────────────────────────────┐
          │ 激活关系数据   补齐英文Prompt   多智能体评审  │
          │ 种子锁定       跨镜上下文      角色生命周期   │
          │ 4帧全利用      多集资产分析    帧插值        │
          │ 参数可配置     叙事丰富阶段    音频生成      │
          │ 前一镜尾帧注入 补齐Negative    质量自动评分   │
          │ 中英混合对齐   LXT资产参考图    变体筛选       │
          │ 道具模型修正    body_proportion修复              │
          │ 下部身体描述                                     │
          │ 画风一致性                                       │
          └─────────────────────────────────────────────┘
```

### 7.5 实测发现的关键问题

#### 问题 1：body_proportion & species_traits 数据丢失 ⚠️ HIGH

- **现象**：LLM 分析完整生成了这些字段，但 DB 持久化后为 `undefined`
- **影响**：角色描述 prompt 缺少体格测量数据，头身比精确控制退化
- **根因推测**：`confirm-profile/route.ts` 中前端传回的 profileData 可能未包含 LLM 分析生成的全部字段，覆盖了完整数据

#### 问题 2：非人类角色下部身体描述系统性缺失 ⚠️ MEDIUM

- **现象**：猪哥哥描述缺少脚蹄/尾巴/下摆/露蹄声明
- **影响**：AI 可能"补全"人类特征（鞋子、裤子），破坏物种一致性
- **建议**：在 LLM prompt 中将⑦下身和⑧配饰标记为必填区

#### 问题 3：LLM 风格推断与用户选择不一致 ⚠️ LOW

- **现象**：用户选 `realistic`，但 LLM 推断为 `Q版卡通风格`
- **建议**：显式将用户 artStyle 传入 `buildDescriptionPrompt`

#### 问题 4：道具提取为 0 ⚠️ LOW

- **现象**：`{"props": []}`，对三只小猪故事合理但可能过于保守
